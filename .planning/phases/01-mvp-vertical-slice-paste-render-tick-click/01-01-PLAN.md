# Plan 01-01: Agenda Parser Fixture and IR

<objective>
Create the single-file app shell with the inlined Markdown parser boundary and a fixture-backed `parseAgenda(markdown)` pipeline that produces a stable agenda IR for title slide, agenda items, and per-person sub-states.
</objective>

<context>
Phase 1 starts with parser correctness because every later renderer, sidebar, and timer feature depends on the same agenda shape. Use `marked.lexer()` for Markdown tokenization, then apply agenda-specific semantics in small helpers. Keep this pure and testable from DevTools.
</context>

<tasks>
<task id="1" type="implementation" files="meeting-manager.html">
<action>Create `meeting-manager.html` with classic inline `<script>` blocks, no module script, and section banners for CSS, marked UMD, parser, model, store, renderer, timer, and boot. Add an inlined `marked` v18 UMD block or a clearly isolated placeholder block that is replaced by the UMD code before Phase 1 is complete.</action>
<verify>Open the file directly from disk in Chrome/Edge. The console must not show module/CORS errors.</verify>
<acceptance_criteria>
- File exists at repo root as `meeting-manager.html`.
- No `<script type="module">`.
- No external `<script src>` or CDN runtime dependency.
</acceptance_criteria>
</task>

<task id="2" type="implementation" files="meeting-manager.html">
<action>Implement `normalizeAgendaText(text)` and `parseAgenda(markdown)` as pure functions. Normalization must strip BOM, normalize CRLF to LF, NFC-normalize text, normalize smart quotes where practical, and preserve user-facing names/titles as strings.</action>
<verify>Run normalization against fixture strings with BOM, CRLF, smart quotes, apostrophes, and unicode names.</verify>
<acceptance_criteria>
- No user agenda content is rendered or stored as HTML.
- Names are treated as opaque strings, never object keys derived from display text.
</acceptance_criteria>
</task>

<task id="3" type="implementation" files="meeting-manager.html">
<action>Walk `marked.lexer(markdown)` tokens to parse `Meeting Admins` and `Agenda` sections into an IR: `{ rawMarkdown, admins, items, slides }`. Parse numbered agenda items for title, attributions, duration, optional deadline, and optional `max N min per person` sub-states.</action>
<verify>Each fixture produces deterministic JSON in DevTools with a title slide plus agenda slides.</verify>
<acceptance_criteria>
- `Meeting Master`, `Notetaker`, and extra admin roles parse from bullet lines.
- Agenda item order follows Markdown order, not user-supplied number sorting.
- Plain items get one timer key; per-person items get one sub-state per attribution.
</acceptance_criteria>
</task>

<task id="4" type="implementation" files="meeting-manager.html">
<action>Add inline parser fixtures covering at least 10 variants: em dash, en dash, double dash, single hyphen with duration lookahead, decimals, missing deadline, multiple attributions, no attribution, max-per-person, BOM/CRLF/smart quotes, and unicode/apostrophe names. Add `runParserSmokeTests()` that logs pass/fail summaries without needing a build step.</action>
<verify>Calling `runParserSmokeTests()` in DevTools reports all fixtures passing and no `NaN`, `undefined`, or `[object Object]` in derived display fields.</verify>
<acceptance_criteria>
- Fixture smoke test is in the single HTML file.
- Parser failures return structured errors with line references where possible, not thrown exceptions that white-screen the app.
</acceptance_criteria>
</task>
</tasks>

<verification>
- Open `meeting-manager.html` from disk.
- In DevTools, run `runParserSmokeTests()`.
- Paste the README agenda into `parseAgenda(...)` manually and inspect the returned title slide, agenda items, sub-states, deadlines, and durations.
</verification>

<must_haves>
- Parser uses `marked.lexer()` tokens for Markdown structure.
- Duration parsing supports `min`, `mins`, `minutes`, `m`, and decimals.
- Separator parsing supports `—`, `–`, `--`, and single `-` only when followed by a duration.
- Parser shape separates slide index from agenda item number from timer key.
</must_haves>

- PARSE-01 through PARSE-10 are covered by code and fixtures.
- The IR is stable enough for Store/Renderer work in Plan 01-02.
</success_criteria>
