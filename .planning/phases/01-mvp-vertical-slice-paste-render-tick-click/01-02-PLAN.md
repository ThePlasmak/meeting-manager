# Plan 01-02: Static Presenter Shell and Sidebar Rendering

<objective>
Turn the parser IR into a visible presenter: paste markdown, click Load, render title/agenda slides, and show a fixed clickable agenda sidebar with correct numbering and sub-state expansion.
</objective>

<context>
This plan establishes the app's one-way data flow before adding timer behavior. The renderer is the only DOM writer; sidebar, stage, and timer bar are persistent regions.
</context>

<tasks>
<task id="1" type="implementation" files="meeting-manager.html">
<action>Implement the base HTML/CSS layout: left sidebar around 25% width, main slide stage around 75%, bottom timer bar region, and paste/load surface. Use system font stack, generous whitespace, no shadows by default, and stable grid dimensions.</action>
<verify>Open the file from disk at 100%, 125%, and 150% zoom. Layout must not overlap or require horizontal scrolling for normal content.</verify>
<acceptance_criteria>
- Sidebar remains visible.
- Stage reads as Marp-style: large title, restrained accent, clean whitespace.
- Timer bar region exists even before timer behavior.
</acceptance_criteria>
</task>

<task id="2" type="implementation" files="meeting-manager.html">
<action>Implement a single store with `state`, `dispatch(action)`, `reduce(state, action)`, and `subscribe(fn)`. Add actions for `LOAD_AGENDA`, `JUMP_TO`, and renderer invalidation. Do not wire persistence yet.</action>
<verify>Loading an agenda mutates only store state; renderer updates through subscription.</verify>
<acceptance_criteria>
- No module besides the reducer mutates shared app state.
- Renderer reads state but does not mutate it.
</acceptance_criteria>
</task>

<task id="3" type="implementation" files="meeting-manager.html">
<action>Implement `AgendaModel` selectors for `getCurrentSlide`, `getCurrentTimerKey`, `getSidebarRows`, `hasSubStates`, and `getSubStateLabel`. Keep title slide and agenda item numbering separate.</action>
<verify>Sidebar item 1 always maps to first agenda item, not title slide.</verify>
<acceptance_criteria>
- Title slide is slide index 0.
- Agenda item labels are 1..N.
- Sub-items render as 1a, 1b, 1c.
</acceptance_criteria>
</task>

<task id="4" type="implementation" files="meeting-manager.html">
<action>Implement region-scoped rendering: `renderSidebar`, `renderStage`, and `renderTimerBarShell`. Sidebar should rebuild on agenda load but only update active/completed classes on cursor changes. Stage must render title slide, normal agenda item slides, and per-person sub-state slides.</action>
<verify>Load an agenda with a max-per-person item; click through sidebar items and inspect active highlights and sub-state labels.</verify>
<acceptance_criteria>
- Slide 1/title slide shows meeting title/admin roles and no active countdown.
- Normal agenda slides show title, attribution subtitle, and deadline badge when present.
- Per-person sub-state slides show current person's name and duration as primary title, with parent agenda title as eyebrow.
- Current sidebar item/sub-item is highlighted.
- Non-current sub-state groups default collapsed except current item expanded.
</acceptance_criteria>
</task>
</tasks>

<verification>
- Paste the README agenda and click Load.
- Confirm title slide plus agenda slides render.
- Confirm sidebar numbering is agenda-based and sub-items expand for max-per-person.
- Confirm sidebar active state changes without resetting scroll position.
</verification>

<must_haves>
- `textContent` or safe DOM node creation is used for user agenda content.
- Store/reducer exists before timer code.
- Sidebar and stage are separate renderer regions.
</must_haves>

<success_criteria>
- LOAD-03, SLIDE-01 through SLIDE-05, and SIDE-01/SIDE-02/SIDE-03/SIDE-05/SIDE-06 are visibly satisfied.
- The app has a stable state/render foundation for Plan 01-03 timer behavior.
</success_criteria>
